# Zoom录音转录与会议纪要生成器

帮助用户将Zoom录制的音频转换为结构化的会议纪要文档。

## 使用方式

```bash
/meeting-minutes /path/to/zoom_recording.m4a
/meeting-minutes /path/to/zoom_recording.m4a --notes /path/to/notes.md
```

## 工作流程

### 步骤 1：验证输入文件

检查用户提供的音频文件是否存在：
- 支持 `.m4a`、`.mp3`、`.wav`、`.mp4` 格式
- 如果提供了人工笔记文件，读取其内容

**输出**：文件信息摘要（文件大小、音频时长）

### 步骤 2：调用MLX-Whisper转录

使用MLX-Whisper进行GPU加速转录（利用Apple M1/M2的Metal GPU）。

**模型选择**：
- 默认使用 `turbo`（large-v3-turbo，最佳质量 + 速度平衡）
- 可选：`small`、`medium`、`large-v3`

### 步骤 3：读取人工笔记（如有）

如果用户提供了 `--notes` 参数，读取笔记内容用于辅助生成纪要。

### 步骤 4：生成详细会议纪要

**重要：纪要必须完整、详细，不能过于简略。**

基于转录内容和人工笔记，生成结构化的会议纪要。**要求覆盖所有主要议题，每个议题都要详细展开。**

#### 详细程度要求

**必须做到**：
1. **覆盖所有主要议题**：不要遗漏任何重要讨论
2. **每个观点都要展开**：不能只写"讨论了XX"，要写清楚"谁说了什么，为什么这么说"
3. **保留关键细节**：具体的数据、案例、比喻都要记录
4. **记录分歧和争议**：不同观点都要呈现
5. **结论要明确**：不能模糊，要清楚地记录最终决定

**避免**：
- ❌ 过于简略的总结（如"讨论了产品方向"）
- ❌ 遗漏重要观点
- ❌ 只记录结论不记录过程
- ❌ 表格只有框架没有内容

### 步骤 5：保存到Google Docs

使用Google Docs API创建真正的表格。

**表格填充原理**：
```
关键：从后往前填充单元格（从最后一行到第一行），
这样前面的索引不会变化。
```

### 步骤 6：报告结果

告知用户：
1. 转录文件路径（本地备份）
2. Google Docs 链接
3. 会议纪要简要摘要（5个要点）

| 对比项 | OpenAI Whisper (CPU) | MLX-Whisper (Metal GPU) |
|--------|---------------------|------------------------|
| 处理速度 | ~0.87x 实时 | ~10-20x 实时 |
| 56分钟音频耗时 | ~65 分钟 | ~3-6 分钟 |
| 加速比 | 基准 | **快 12-20 倍** |

## 配置要求

### Google Docs API 配置

1. 访问 https://console.cloud.google.com/
2. 创建项目并启用 Google Docs API
3. 配置 OAuth 同意屏幕
4. 创建 OAuth 2.0 客户端 ID（桌面应用）
5. 下载JSON保存到 `~/.credentials/google_client_secret.json`

## 参数说明

| 参数 | 说明 | 默认值 |
|------|------|--------|
| `audio` | 音频文件路径 | 必填 |
| `--notes` | 人工笔记文件路径 | 可选 |
| `--model` | Whisper模型 | turbo |
| `--no-upload` | 不上传Google Docs | False |
| `--output-dir` | 输出目录 | 音频文件同目录 |
